# Creature Card System v2 — Data Model + Layout Architecture Investigation

## 1. Executive summary

This investigation reviews the current Sanctuary creature-card stack and proposes a **forward-compatible v2 card metadata architecture** that can support both:

- today’s simple Sanctuary grid/front card experience, and
- a future premium full-screen collectible card with richer narrative metadata.

Recommendation: adopt a **new additive metadata layer** (v2) that composes existing `CreatureDefinition` + `CreatureCardMetadata` data rather than replacing them immediately. Keep current card rendering contracts stable, then phase in v2 by introducing selectors/adapters that can project:

- `simpleFront` contract (grid + lightweight card surfaces),
- `fullFront` contract (header/art/ability/tags/collection),
- `backside` contract (lore/memories/unlock/progression/deep profile).

This preserves current behavior while de-risking future expansion into variants/stages/event editions.

## 2. Current card/data architecture

## 2.1 Canonical species source (`creatureCatalog`)

`CreatureDefinition` is the current baseline species schema with:

- `id`, `imageKey`, `name`
- `tier` (`common | rare | mythic`)
- `habitat`, `affinity`, `shipZone`

This is currently sufficient for selection, display identity, and core Sanctuary labels, but not sufficient for full collectible card semantics (dex numbering, passive/move structure, strengths/weaknesses, lore, variants, etc.).

## 2.2 Card-presentational metadata (`creatureCardCatalog`)

`CreatureCardMetadata` currently adds display-first fields:

- `displayName`, `shortTitle`, `rarityLabel`
- `flavorQuote`
- `passiveName`, `passiveText`
- `powerLabel`, `statLine`
- `theme` (`tier`, `shipZone`, `accent`, `template`)

The catalog is generated by mapping each `CreatureDefinition` through curated metadata with fallback generation. This gives broad card coverage while tolerating missing curated entries.

## 2.3 UI contracts (today)

### `CreatureCard` (richer front card)

Uses metadata + art manifest to render:

- top bar: rarity/accent, name, power/state badges
- art window
- identity row (short title + stat line)
- ability block (passive)
- flavor quote

### `CreatureGridCard` (sanctuary-minimal)

Very lightweight tile with:

- art/silhouette
- rarity stars
- active marker
- locked label

No deep metadata is required here (good for current design direction).

### `CreatureHatchRevealModal`

Uses a minimal reveal contract:

- creature name
- rarity + star string
- creature score
- art
- action buttons

## 2.4 Sanctuary adapter (`creatureSanctuaryAdapter`)

`buildCreatureSanctuaryGalleryModel` creates read models for gallery usage, including:

- discovery/copies
- bond level/xp
- active companion state
- tier counts and summary
- egg-status summary

This adapter is a natural place to compute **collection progress outputs** (e.g., `08/45`) for card surfaces without polluting base species metadata.

## 2.5 Asset manifest (`creatureImageManifest`)

Current manifest resolves:

- cutout assets (webp/png)
- tier frame
- affinity/zone background
- silhouette placeholder
- emoji fallback

This is solid for current front art composition and can be extended for layout-specific layers (type icons, rarity badges, edition stamps, backside motifs) through additive keys.

## 3. Current field mapping to v2 card zones

## 3.1 Header zone (name, type icon, creature number)

**Current coverage**
- name: `CreatureDefinition.name` / `CreatureCardMetadata.displayName`
- type-ish label: partially via `affinity`, `shipZone`, `theme.accent`

**Missing**
- canonical creature number / dex index
- explicit icon token/id for type/class/affinity

## 3.2 Main art stage

**Current coverage**
- `imageKey` + resolved cutout/background/frame
- hatch/grid/card use same core art paths

**Missing**
- stage art variants per card edition/state
- explicit art-direction slots (hero art vs portrait crop vs foil mask)

## 3.3 Rarity / class / affinity strip

**Current coverage**
- rarity: `tier` + `rarityLabel`
- affinity: `CreatureDefinition.affinity`

**Missing**
- distinct class taxonomy (if different from affinity)
- normalized tokenized strip fields

## 3.4 Passive ability + move/ability section

**Current coverage**
- one passive pair (`passiveName`, `passiveText`)

**Missing**
- structured multi-ability list
- per-ability type/category/cost/value formatting

## 3.5 Strength/weakness tags

**Current coverage**
- none (closest signal is `shipZone`/`affinity`)

**Missing**
- explicit strengths/weaknesses arrays
- controlled vocabulary/tag ids

## 3.6 Flavor quote

**Current coverage**
- `flavorQuote`

**Missing**
- author/source/voice attribution (optional future)

## 3.7 Collection progress

**Current coverage**
- available from adapter summary totals/discovered counts

**Missing**
- explicit card-level projection contract for “08/45” text

## 3.8 Backside/deep metadata

**Current coverage**
- partial via habitat/affinity + runtime bond data

**Missing**
- origin lore, unlock source, favorite foods, habitat story, event history, stage notes, synergy tags

## 4. Proposed v2 data model

Use a compositional model with three layers:

1. **Species Core** (existing `CreatureDefinition`, unchanged in first slice)
2. **Card Presentation v2** (new metadata layer)
3. **Player Projection** (runtime-derived fields such as collection progress, bond progression)

Proposed shape (illustrative TypeScript):

```ts
interface CreatureCardV2Record {
  creatureId: string;
  version: 2;

  identity: {
    dexNumber?: number;              // optional until catalog-wide assignment
    displayName: string;
    rarity: 'common' | 'rare' | 'mythic';
    classLabel?: string;             // optional taxonomy separate from affinity
    affinityLabel: string;
    typeIconKey?: string;            // maps to manifest/icon assets
  };

  frontSimple: {
    title?: string;                  // optional override, default displayName
    subtitle?: string;               // compact surface (grid/detail-list)
    rarityBadgeMode?: 'stars' | 'label';
  };

  frontFull: {
    header: {
      creatureNumberLabel?: string;  // e.g., "No. 008"
      rarityLabel: string;
      classLabel?: string;
      affinityLabel: string;
    };
    art: {
      artKey?: string;               // defaults to CreatureDefinition.imageKey
      frameKey?: string;
      backgroundKey?: string;
      stageKey?: string;             // supports future stage-specific art
    };
    passive?: {
      name: string;
      description: string;
    };
    abilities?: Array<{
      id: string;
      name: string;
      description: string;
      valueBadge?: string;           // e.g., "COST 2" / "PWR 12"
      kind?: 'move' | 'skill' | 'aura' | 'utility';
    }>;
    tags?: {
      strengths?: string[];
      weaknesses?: string[];
    };
    flavor?: {
      quote?: string;
    };
  };

  backside?: {
    originLore?: string;
    unlockSource?: string;
    favoriteFoods?: string[];
    habitatStory?: string;
    synergyTags?: string[];
    eventHistory?: Array<{
      eventId: string;
      label: string;
      note?: string;
      seasonKey?: string;
    }>;
    stageEvolutionNotes?: string[];
  };

  variants?: Array<{
    variantId: string;               // e.g., base, foil, event-lantern-festival
    editionType: 'base' | 'event' | 'seasonal' | 'anniversary' | 'promo';
    stageKey?: string;
    artOverrides?: {
      artKey?: string;
      frameKey?: string;
      backgroundKey?: string;
      badgeKey?: string;
    };
    textOverrides?: {
      flavorQuote?: string;
      passiveName?: string;
      passiveDescription?: string;
    };
    availability?: {
      startAtIso?: string;
      endAtIso?: string;
      sourceLabel?: string;
    };
  }>;
}
```

Key design choices:

- `frontSimple` and `frontFull` are separate, so Sanctuary grid can stay minimal.
- `backside` is optional and can gradually fill without blocking rollout.
- `variants` are overlay-style, avoiding duplication of full base records.
- Player/runtime fields (copies, bond, discovered, progress fraction) remain outside static catalog metadata.

## 5. Simple front card contract

Purpose: Sanctuary grid + compact list surfaces.

Recommended selector output (`CreatureCardSimpleView`):

```ts
interface CreatureCardSimpleView {
  creatureId: string;
  displayName: string;
  rarity: 'common' | 'rare' | 'mythic';
  rarityLabel: string;
  starLabel: string;
  image: {
    cutoutSrc: string;
    silhouetteSrc: string;
    fallbackEmoji: string;
  };
  state: {
    discovered: boolean;
    locked: boolean;
    active: boolean;
  };
  collection: {
    ownedCopies: number;
    progressLabel: string; // e.g., "08/45"
  };
}
```

Notes:

- Keep this intentionally small and cheap.
- Avoid lore/ability clutter on grid tiles.
- Compute `progressLabel` from adapter summary, not static metadata.

This contract maps directly to requested zones while keeping player progression (`progressLabel`) as a read-model concern.

## 7. Backside/deep metadata contract

Purpose: lore/progression profile view (detail modal backside), not grid.

Recommended selector output (`CreatureCardBackView`):

```ts
interface CreatureCardBackView {
  creatureId: string;
  originLore?: string;
  unlockSource?: string;
  favoriteFoods: string[];
  bond: {
    currentLevel: number;
    currentXp: number;
    nextMilestoneLevel?: number;
    notes?: string[];
  };
  habitatStory?: string;
  synergyTags: string[];
  eventHistory: Array<{
    label: string;
    note?: string;
  }>;
  stageEvolutionNotes: string[];
}
```

Runtime bond values should come from collection state; static notes/lore come from v2 metadata.

## 8. Variant/stage/event-edition strategy

Recommended strategy: **base record + overlay variants**.

- Base record defines stable identity + default front/back fields.
- Variant overlay defines only changed fields (art/text/badges/availability).
- Stage progression is represented as `stageKey` + optional stage art/text overrides.

Benefits:

- avoids duplicating full card payloads for seasonal/event editions,
- supports future “same creature, different edition” presentation,
- keeps legacy/simple surfaces stable by resolving a selected variant at selector time.

Variant resolution order (suggested):
1. explicit requested variant id,
2. active event-eligible variant,
3. base variant.

## 9. Risks and constraints

## 9.1 Overloading `CreatureCardMetadata`

If v2 fields are stuffed directly into current `CreatureCardMetadata`, it will become a mixed legacy/new shape with high optionality and weak semantics. This increases UI coupling risk.

## 9.2 Taxonomy drift (affinity vs class vs type)

Current model has `affinity` and `shipZone`; “class/type” semantics are not yet canonical. Introduce explicit labels/keys in v2 to avoid stringly-typed inference in UI.

## 9.3 Asset-key sprawl

Without manifest conventions for icon/badge/edition assets, variants can produce inconsistent paths. Keep all new keys manifest-driven.

## 9.4 Data provenance split

Some fields are static catalog metadata; others are runtime (bond/progress/active). Mixing these in one persistence layer creates migration complexity. Keep clear separation.

## 9.5 Scope constraints for this phase

Per current constraints, no schema/UI/gameplay logic changes are made now. This report only defines structure and migration-safe slicing.

## 10. Recommended PR slices

## Slice 1 — Type-only model introduction (no behavior change)

- Add new `CreatureCardV2Record` type definitions and documentation comments.
- No component usage changes.
- No runtime/store/schema changes.

## Slice 2 — Static adapter bridge

- Add pure adapter(s) that map existing `CreatureDefinition + CreatureCardMetadata` into provisional v2-compatible `simple/full/back` view objects.
- Fill unsupported fields as `undefined`/empty arrays.
- No UI changes.

## Slice 3 — Collection progress projection utility

- Add selector utility to produce standardized collection progress labels (`discovered/total`) for any card surface.
- Consume existing sanctuary adapter summaries only.

## Slice 4 — Variant-ready manifest contracts

- Add manifest key contracts/interfaces for type icons, variant badges, and stage art keys (types only or pure resolver stubs).
- No new assets required in same PR.

## Slice 5 — Gradual data enrichment pipeline

- Add optional content authoring files/JSON schema for deep metadata and variant overlays.
- Keep optional and non-blocking; preserve fallback behavior.

## Slice 6 — UI adoption (future, separate from this investigation)

- First adopt simple/full/back view contracts in detail modal surfaces.
- Keep Sanctuary grid minimal by design.

---

## Recommendation: additive vs separate layer

**Recommend a separate new metadata layer (`CreatureCardV2Record`) that is additive to existing systems, not a direct mutation of `CreatureCardMetadata`.**

Rationale:

- preserves current rendering contracts,
- enables controlled migration via adapters,
- avoids destabilizing existing curated metadata/fallback logic,
- keeps long-term rich-card fields semantically grouped,
- supports variant/stage/event overlays cleanly.
